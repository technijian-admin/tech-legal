The engine's ONLY throwers are `QbConnectionManager` (`QbException` / `QbBusyException` / `QbTimeoutException`) and `ArgReader` / the ops (`ArgumentException` on bad/missing args) and `QbXmlParser` / `QbWriteDetector` (`QbXmlParseException` on malformed XML). Never map `QbStatus.IsError` to an HTTP error.

**API-05 interpretation (decided — do NOT re-litigate):** `/api/ops/{op}` returns `200 { "op": "<name>", "result": <op's returned dict> }`. Verbatim raw qbXML for arbitrary requests is `/api/qbxml`'s job; **do NOT refactor `ReadOpBase` to carry the raw response** (that's a shipped Phase-4 surface). The lighter reading is the one this plan adopts.

**OpRegistry for Phase 7 (decided — do NOT pre-build):** `OpRegistry` takes `IEnumerable<IReadOp>` in Phase 5. Do NOT widen `IReadOp` / introduce a new op interface / pre-wire write-op dispatch — leave that call to Phase 7's planner.
