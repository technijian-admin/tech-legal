<objective>
Implement the v1 QuickBooks WRITE operations (WRITE-03..07) as thin `WriteOpBase` subclasses riding the Phase-6 write-safety / dry-run / audit / `AllowWrites`-gate machinery: seven `create_*` ops (`create_customer`, `create_vendor`, `create_invoice`, `create_bill`, `create_check`, `receive_payment`, `create_journal_entry`) plus ONE generic `mod` op (`{entity, ref, fields}`). Each `create_*` op implements a pure `BuildRequest(args) -> qbXML` and overrides `DryRunAsync` for real pre-flight; the inherited `WriteOpBase.RunAsync` does gate -> execute -> parse -> append one audit row. The `mod` op overrides `RunAsync` (read current `*Ret` + its `EditSequence` -> merge `fields` over it -> strip read-only/computed children -> build `<{Entity}ModRq>` with the just-read `EditSequence` -> execute -> parse -> audit), never retries a stale `EditSequence` (3200), and its `BuildRequest` throws unless the record was already resolved. `WriteOpBase` gains `FetchByNameAsync` / `FetchCurrentAsync` (filling the existing `// TODO(Phase 7)` slot); `WriteOpHelpers` (new) holds `MultiCurrencyGuard.Reject` + ref/address/line-item `XElement` builders; `ArgReader` gains `List` / `Decimal` / `RequiredString`. All 8 ops are registered as `IReadOp` singletons in `Program.cs` so the existing `OpRegistry`/`OpsEndpoints`/`POST /api/ops/{op}`/`POST /api/ops/{op}/dryrun`/audit/3-layer-gate handle them with no endpoint changes.

Purpose: This is the "writes" deliverable of the project's core value — Claude can create/update a QuickBooks transaction, but only after an explicit dry-run-and-confirm step, with every executed write in the immutable audit log.
Output: 11 new/extended `.cs` files under `Qb.Com/Qb/Ops/`, the `Program.cs` DI lines, 6 new test files + 2 extended (`OpRegistrationTests`, `OpTestHarness`), 11 new constructed qbXML fixtures, and the standard end-of-phase docs (SUMMARY, REQUIREMENTS WRITE-03..07 ticked, ROADMAP Phase-7 boxes, STATE).
