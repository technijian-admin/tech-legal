<objective>
Build the Phase 1 skeleton for the QuickBooks Enterprise Direct-SDK accounting service: a .NET 8 solution under `tech-legal/quickbooks/QbConnectService/` with three projects (ASP.NET Core host, isolated COM adapter, xUnit tests), the mockable `IRequestProcessor` seam, a `FakeRequestProcessor`, a stubbed `RealRequestProcessor` (every method `throw new NotImplementedException(...)`), a tri-mode host (`AddWindowsService()`), the x86 single-file build settings, tests that make SVC-02/03/04 machine-verifiable, and a GitHub Actions CI workflow.

Purpose: This phase exists so the entire stack above the COM boundary is buildable and testable on a Windows box with NO QuickBooks SDK installed. Every later phase (2–8) builds on this seam; only Phase 9 touches the live QuickBooks host. Get the seam and the build hygiene right here and the rest of the project compounds cleanly.

Output: A solution + three .csproj files + `IRequestProcessor.cs` + `QBXMLRP2Lib.cs` interop stub + `RealRequestProcessor.cs` stub + `Program.cs`/`Worker.cs` + `FakeRequestProcessor.cs` + three test files + `quickbooks-ci.yml` + appended `.gitignore` + `appsettings.sample.json` + two short READMEs.

**SCOPE DISCIPLINE — Phase 1 is JUST the skeleton.** Do NOT build: `QbConnectionManager`, `QbErrors`, the STA worker thread, retry logic, `QbXmlBuilder`/`QbXmlParser`, any report parser, any read or write op, any controller, Kestrel HTTPS binding, bearer auth, the `AllowWrites` gate, the audit log, the Python client, the Claude skill, or the deploy scripts. Those are Phases 2–9. If a task in this plan seems to invite any of that, stop — it's out of scope.
</objective>
